Veltrax — Architecture Local‑first • Private by default

Overview

Veltrax is a private, local‑first AI platform that runs on hardware you control. It ingests data from cloud vendors and on‑prem systems, stages it in a secured DMZ, and processes everything inside the Veltrax Core (Apps + Postgres + Local LLMs). Results and audit trails flow to employee workstations—without SaaS LLMs.

Design goals

  • Zero external LLMs by default. All inference local.
  • Proven business impact. Every pipeline feeds ROI & audit.
  • Defense‑in‑depth. DMZ boundary, signed updates, RBAC, immutable logs.

Works for banking (Finex), retail (ÉCHELON), and legal (Verdict).

Veltrax Core

Runs on customer hardware. Everything happens inside your network.

  • Apps: Finex (banking), ÉCHELON (retail), Verdict (legal).
  • Storage: Postgres with PITR, encryption at rest.
  • Models: Local LLMs & task‑specific models, quantized for the target GPU/CPU.
  • Auth: SSO/OIDC, JWT, and RBAC enforced at the service layer.

DMZ

  • Pulls signed update/data bundles on a schedule (mTLS).
  • Verifies provenance (Sigstore/Cosign). No long‑lived secrets at rest.
  • Transfers across the boundary into the Core via approved jobs.

Air‑gappedDMZ relay Both modes supported.

Data Flows

  1. Cloud vendors → DMZ → Core: vendor exports are signed, staged in DMZ, then applied to the Core after verification.
  2. Local sensitive data → Core: read‑only connectors (JDBC/ODBC/SFTP) ingest operational data without exposing secrets externally.
  3. Core → Workstations: apps served on LAN; all actions are audit‑logged and summarized for ROI.

Security Model

  • Defense‑in‑depth: DMZ boundary, network segmentation, least‑privilege service accounts.
  • mTLS everywhere; certificates rotated automatically.
  • Encryption: AES‑256 at rest, TLS in transit.
  • RBAC & signed release artifacts; optional quorum for updates.

Compliance & Audit

  • Immutable event logs with content hashes.
  • Deterministic pipelines; every prediction is explainable and traceable.
  • Export to CSV/PDF for auditors; JSONL for data teams.

Deployment Options

  • Air‑gapped: media transfer of signed bundles; no direct network egress.
  • DMZ relay: pull over mTLS on a fixed cadence; no inbound ports opened to Core.

Network & Ports

  • DMZ → Core: pull jobs over mTLS (one‑way).
  • Workstations → Core: HTTPS (443) through the LAN with SSO.
  • Source systems → Core: JDBC/ODBC/SFTP as read‑only users.

Signed Updates

  1. Build pipeline produces an SBOM + signed artifacts.
  2. DMZ fetches the release manifest and verifies signatures.
  3. Validated packages are staged and applied to Core with policy checks.

All artifacts are attestable with provenance; rollbacks are deterministic.

Observability

  • Structured event stream (JSONL) per service.
  • Resource metrics (CPU/GPU/mem/IO) with budgets per workload.
  • Trace replay for incident analysis.

ROI & Audit

  • KPIs surfaced per department (savings, fraud prevented, time saved).
  • Every action mapped to a ledger entry with an owner and timestamp.
  • Exports for Finance & Compliance (CSV/PDF).

Reference Sizing

  • 128GB RAM • NVMe 2TB
  • 100–200 concurrent users

Standard

  • 2–4× GPUs (80GB VRAM class)
  • 256–512GB RAM • NVMe 8TB
  • 500–2k concurrent users

Enterprise

  • 8× GPUs (H100/A100 class) + CPU inference tier
  • 1–2TB RAM • NVMe 16TB+
  • Multi‑site HA + DR
